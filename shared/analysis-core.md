# 賽事分析共用契約

所有 `*-analysis` skill 先套用本契約，再套用各項運動的專屬規則。若兩者衝突，以較嚴格且較貼近該運動結算方式的規則為準，並在輸出中說明。

## Codex × agy 協作管線

使用者要求 agy、agy-cli、紅隊審查或模型互審時，同時套用 `../prediction-pipeline/SKILL.md`。下列任一短語都視為明確觸發：

- `啟用 agy 紅隊`
- `用 agy 紅隊審查`
- `交給 agy 複核`
- `啟用雙模型審查`

只有觸發 agy 紅隊時才顯示模型預檢；一般單模型分析不得詢問模型。紅隊觸發後，先讀取 `../prediction-pipeline/references/model-defaults.json` 的預選值，再執行 `agy models` 確認可用選項，向使用者確認「agy 紅隊模型」與「Codex 最終裁決模型／推理強度」。設定檔不得關閉確認；未收到明確確認前不得開始資料收集、主預測或任何模型呼叫。

確認後在背景依下列順序執行，不要要求使用者自行建立 JSON、指定目錄或輸入 CLI 指令：

```text
資料收集器 → input.json → Codex 主預測 → agy 紅隊審查
→ Codex 最終裁決 → 確定性驗證 → Markdown / JSON / YouTube 腳本
```

- 主預測階段只讀 `model-input.json`，不得接觸 `market_data`。
- agy 的輸出是待裁決的審查意見，不得直接當成最終預測。
- 最終 Codex 必須逐項裁決 critical/high finding；市場資訊只能用於機率固定後的價格與 EV 比較。
- 驗證失敗時停止匯出與發布，回到產生錯誤的上游階段重建。
- 未要求協作時維持原本單模型流程，不必額外呼叫 agy。
- 一般對話由目前的 Codex 執行主預測；最終裁決使用者若選「目前會話模型」則仍由目前 Codex 執行，若選 Codex CLI 模型才另啟該模型。

## 資料與時間

- 以台灣時間（UTC+8）解讀「今天」「明天」等相對日期，必要時同列比賽所在地日期。
- 對賽程、名單、傷停、版本、先發、場地、天氣與盤口等易變資訊進行即時查核，不以模型記憶代替。
- 區分已確認事實、可靠報導、推估與未驗證資訊；找不到資料時揭露缺口並降低信心度。
- 重要事實附可追溯來源。來源衝突時依該 skill 的 `references/source-priority.md` 處理。

## 模型與市場分離

依序完成以下階段，不可倒置：

1. 只用賽事因素建立模型機率，例如陣容、版本、對位、近期內容、賽程與場地。
2. 固定模型機率與模型信心度。
3. 才讀取市場價格，計算公允賠率、期望值與可接受價格。

不得用市場賠率、隱含機率、盤口水位或價格變動反推勝負、修改模型機率或調整模型信心度。若市場與模型明顯分歧，只能觸發資料遺漏檢查；只有找到新的賽事事實後才能重建模型。

- `模型勝率`：對結果的機率估計。
- `模型信心度`：資料完整度、名單或先發確定性、樣本相關性與模型內部一致性；不含價格或投注價值。
- `投注價值`：模型機率與市場價格的差距；沒有即時價格時只給公允賠率與價格門檻。

所有賠率使用十進位。不得承諾獲利、鼓勵追損或建議 all-in。

```text
公允賠率 = 100 / 模型機率百分比
EV = 市場十進位賠率 * (模型機率百分比 / 100) - 1
```

## 輸出模式

依使用者需求選擇最小充分輸出，不要把完整模板套到每個問題：

- `quick`：單一問題或使用者要求簡版。直接回答結論、核心證據、主要風險與資料狀態。
- `full`：單場完整賽前分析。讀該 skill 的 `references/output-template.md`，保留所有必要機率與風險欄位。
- `daily-summary`：先盤點指定台灣日期的全部目標賽事，再用精簡表格比較；只對入選場次展開分析。
- `postmortem`：讀該 skill 的賽後校準 reference；重建賽前資訊集與實際事件，不用結果倒推當時必然可知。

使用者未指定時，單場分析預設 `full`，單一追問預設 `quick`，整日賽程預設 `daily-summary`。使用者明確要求的長度與欄位優先。

所有對外的賽事總表或簡表都必須保留 `模型信心度` 欄位，包括 `full`、`daily-summary`、多場比較與 `postmortem` 的原預測比較表。使用者要求更短時仍不得刪除；資料不足則填 `N/A（原因）`。模型信心度必須使用百分比並與勝率分欄，不得以勝率、星號、火焰符號或「高／中／低」取代。

## 所有新預測的最終輸出契約

本節適用於每個 `*-analysis` skill、單模型分析、agy 紅隊流程及 `prediction-pipeline` 匯出，不因運動、賽制或輸出模式而省略。

1. 每份包含新預測的輸出（`quick`、`full`、`daily-summary` 或自訂格式）必須且只能有一個表格形式的 `簡表總結`。
2. 先完成所有場次分析，再輸出來源、風險、資料缺口、盤口說明與免責文字；最後才輸出 `簡表總結`。多場報告不得在場次之間插入簡表，而要把所有場次合併到同一張置底表格。
3. 簡表欄位依各運動的 `references/output-template.md` 決定，但至少保留比賽、主要預測、模型信心度、建議與核心風險；不得以一段純文字的「預測總結」取代表格。
4. `簡表總結` 是最後一個實質段落。表格之後不得再新增分析、來源、風險、免責說明或附註。
5. 表格後只允許加入模型揭露，且模型揭露必須是全文最後一個非空白行：

```text
預測使用模型：<執行環境提供的模型名稱>
```

- 揭露實際產生最終預測報告的模型；雙模型／紅隊流程揭露最終裁決模型，不以紅隊模型取代。
- 使用系統或執行環境明確提供的完整模型 ID 與推理強度；兩者皆已知時以空格串接，例如 `gpt-5.6-sol high`。
- 已知完整模型 ID 與推理強度時，禁止縮寫成 `GPT-5`、`GPT5` 等家族名稱。只有執行環境確實只提供模型家族時，才可忠實填寫該家族，不得猜測子版本、快照或推理強度。
- 若執行環境完全未提供模型名稱，填 `預測使用模型：執行環境未提供`，不得自行猜測。
- 此欄揭露的是產生分析的 GPT 模型，不是勝率模型、統計方法或資料來源，也不得取代 `模型信心度`。
- 純賽後檢討若沒有產生新預測，不強制加入簡表與模型揭露；只要產生更新後的新預測，就必須完整套用本節。

送出或匯出前必須檢查：`簡表總結` 只出現一次、它位於所有實質內容之後、最後一個非空白行符合 `預測使用模型：...`。任一條件不符即先修正，不得發布。

## 機率與結算檢查

- 互斥且完備的結果分布合計為 100%。
- 系列賽勝率等於該隊所有獲勝精確比分的機率總和。
- 「至少一局／一圖」與「被橫掃」互補。
- 有和局或走盤可能時使用三項分布，例如 `大分 + 小分 + 走盤 = 100%`；若只報兩側，標明是排除走盤後的條件機率。
- 信心度不可代替勝率，也不需要和任何機率相加為 100%。
- 不輸出無資料依據的假精準數字；四捨五入造成的誤差需保持在 0.2 個百分點內。

完整分析在送出前，把數字整理成 JSON 並執行：

```bash
node shared/validate_probabilities.mjs <checks.json>
```

`checks.json` 使用下列結構；`equal` 的 `left` 應等於 `right` 陣列總和：

```json
{
  "checks": [
    { "type": "sum", "name": "精確比分", "values": [30, 35, 20, 15] },
    { "type": "equal", "name": "A 勝率", "left": 65, "right": [30, 35] },
    { "type": "complement", "name": "至少一圖與被橫掃", "values": [85, 15] },
    { "type": "fair_odds", "name": "A 公允賠率", "probability": 65, "fairOdds": 1.5385 },
    { "type": "ev", "name": "A EV", "probability": 65, "marketOdds": 1.7, "ev": 0.105 }
  ]
}
```

若驗證失敗，先修正數字或結算定義再輸出；不可忽略錯誤。

## 行動與外部寫入

- 查資料、分析、建立本地暫存檔及執行非破壞性驗證屬於安全的範圍內動作。
- 寫入 Notion 或其他外部服務屬於外部寫入。使用者在當前請求明確要求同步時可以執行。
- 僅有 `NOTION_AUTO_PUBLISH=1` 而當前請求未提及發布時，先完成分析與本地匯出檔，取得使用者確認後才發布。
- 賽後測試、回測與 skill 驗證預設不得對外發布。
