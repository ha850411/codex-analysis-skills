# League of Legends 賽事深度分析技能

當使用者要求 League of Legends 英雄聯盟電競賽事分析、賽前預測、當日賽程分析、選手名單核對、版本影響、賠率評估，或 LCK、LPL、LCP、LEC、LCS、EWC、德瑪西亞盃、LCK CL 等相關 LoL 賽事的投注角度時，使用此技能。

預設語言：繁體中文，除非使用者另有要求。
預設時區：台灣時間 UTC+8。使用者提到「今天」「明天」「等一下」等相對日期時，一律以台灣時間解讀。
賠率輸出格式：所有報告中的市場賠率與公允賠率一律使用十進位（decimal odds，例如 `1.85`）。不得輸出美式、分數、香港盤、馬來盤、印尼盤或其他格式；來源若不是十進位，只能內部轉換後再比較。

勝負判斷限制：絕對不能用賠率、盤口水位、隱含機率或賠率變動來分析勝負、賽果方向、反推勝率或校準勝負判斷。勝負與賽果機率只能由名單、版本、BP、對線、物件控制、團戰、近期內容、賽程與選邊等賽事因素推導；賠率只能在模型機率完成後用於 EV、價格門檻與是否值得下注。

## 不可妥協的資料來源規則

1. **賽程主要來源：** 只要可行，目標賽事賽程以 `lol.fandom.com` 為準。
2. **名單主要來源：** 隊伍成員與先發名單必須核對 `lol.fandom.com`。其他名單來源若與 LoL Fandom 衝突，不要採信其他來源。
3. **版本主要來源：** 查看 `lol.fandom.com` 上相關比賽或賽事頁面，確認該比賽日或階段使用的版本。分析版本如何改變 BP 優先級、線路對位、物件節奏、英雄池與選邊價值。
4. **數據與歷史對戰：** 可參考 `esports.op.gg/schedules`、近期比賽頁、局內數據與其他可信數據來源，但不可盲從。只看比分不足以支撐分析。
5. **近期 VOD 複盤：** 可取得時，檢視或引用正確賽區來源的近期 VOD：
   - LCK: `https://www.youtube.com/@LCKglobal`
   - LPL: `https://space.bilibili.com/50329118`
   - LCP: `https://www.youtube.com/@lolesportstw/videos`
   - LEC: `https://www.youtube.com/@LEC`
   - LCS: `https://www.youtube.com/@LCS`
6. **不可捏造事實：** 如果來源無法存取，或先發名單無法驗證，必須清楚說明、降低信心度，且不可把推測寫成已確認事實。

## 必要工作流程

1. **確認比賽範圍**
   - 釐清賽區、日期、對戰組合、賽制、台灣開賽時間、賽事階段與版本。
   - 如果使用者問「今天某賽區」，找出台灣時間該日期的全部比賽。

2. **驗證雙方隊伍**
   - 從 LoL Fandom 取得雙方隊名。
   - 在 LoL Fandom 檢查目前陣容與可能/已確認先發。
   - 回答中必須列出雙方先發，讓使用者可以核對。
   - 名單依位置列出：Top / Jungle / Mid / ADC / Support。替補只在相關時列入。

3. **近期狀態分析**
   - 分析約過去一個月的比賽。
   - 最近 1–2 週的內容優先於較早比賽。
   - 可取得時，加入近期對戰紀錄。
   - 比較交手對手強度，不只看勝負。
   - 重點放在比賽內容：BP 邏輯、對線期、打野路線、物件佈置、邊線管理、團戰執行、巴龍/小龍控制、視野紀律、前期滾雪球穩定性、被翻盤風險，以及局間調整能力。

4. **版本與 Meta 影響**
   - 確認比賽版本。
   - 說明版本對哪些英雄池、線路、隊伍風格與 ban/pick 有利或不利。
   - 相關時補充紅藍方選邊影響。

5. **影片輔助判斷**
   - 可行時使用 VOD、官方精華或賽後回顧。
   - 對每支隊伍提出近期比賽中的具體觀察，例如前期打野規劃、換線應對、物件佈置時間點、中期節奏與收尾能力。
   - 如果無法完整複盤 VOD，必須明確說明限制，並提高比賽數據與可取得回顧內容的權重。

6. **預測模型**
   - 預測勝方與精確系列賽比分。
   - 提供所有合理精確比分的機率。
     - BO3：2–0、2–1、1–2、0–2。
     - BO5：3–0、3–1、3–2、2–3、1–3、0–3。
   - 提供雙方整體勝率。
   - 提供雙方各自至少贏一局的機率。
   - 以百分比提供信心度，並說明信心高/中/低的理由。
   - 明確區分模型機率與投注賠率。
   - 產出推薦前，先依 `references/recommendation-gates.md` 檢查打滿、地圖讓分、至少贏一局與精確比分是否有足夠證據。
   - 若推薦 BO5 3-2、地圖大分、+1.5 或「雙方各取一局」，必須列出雙方各自的具體取圖路徑；若只能靠 H2H、名氣或「強隊不會被橫掃」這類直覺，降低信心、提高橫掃機率，並避免推薦該市場。
   - 若模型勝率與市場熱門 implied probability 相差 10 個百分點以上，先做市場分歧檢查：確認是否漏看版本答案、局間調整、選邊、賽程或 VOD 訊號；不能只因獨贏有價格，就自動推導到敗方 +1.5 或至少贏一局。
   - 若主推或高順位推薦是弱方 +1.5、Over 3.5、雙方皆拿圖，必須做「讓分/爆冷一致性檢查」：弱方若有兩條以上可重複取圖路徑，且具備近期 H2H、休息/準備、版本適性或核心 carry 優勢之一，弱方勝率不得被壓成純尾端；熱門方勝率超過 60% 時需明確說明弱方為何只能拿圖、不能收下系列賽。

7. **賠率與投注建議**
   - 報告中的市場賠率、公允賠率與價格門檻只可使用十進位格式。
   - 如果可取得 Stake 賠率，或使用者提供賠率，將市場賠率與模型公允賠率比較。
   - 如果 Stake 無法存取或缺少賠率，先給出模型公允賠率；完成當前分析後，再請使用者提供即時賠率以精確比較 EV。
   - 可行時討論可玩的市場，例如獨贏、讓分、精確比分、地圖總數、首局，或雙方各贏一局。
   - 地圖讓分、地圖大分與「至少贏一局」屬於高誤差市場，必須通過推薦閘門；未通過時即使賠率看似便宜，也標記為不碰或只看 live。
   - 敗方 +1.5、地圖大分與「敗方至少一局」不得用「價格型」當唯一理由；必須同時列出敗方兩條可執行取圖路徑與熱門方橫掃為何受限。若做不到，獨贏可小注但地圖盤不推薦。
   - 如果最佳投注角度其實是弱方 +1.5 或地圖大分，回答中的主結論不要寫成熱門方穩勝；應改寫為「熱門小優但系列賽接近」或「弱方具備 live upset」，並避免給熱門方高信心 ML。
   - **不可**承諾獲利或保證結果。
   - **不可**建議失控式 all-in 投注。即使判斷非常強，也只能標示為「高信心」，並以資金控管限制注碼。
   - 建議注碼語言：
     - 小注傾向：0.25–0.5u
     - 正常可打：0.5–1u
     - 強勢可打：1–2u
     - 避開：不下注 / 等更好價格 / 只看 live

8. **風險段落**
   - 必須包含潛在風險：名單未確認、版本理解不確定、選邊、賽程疲勞、移動/延遲、動機、替補、BP 陷阱、前期風格波動、物件硬拚傾向與賠率變動。

9. **最終表格**
   - 必須以簡潔表格收尾，彙整：
     - 比賽
     - 已確認/可能先發
     - 版本
     - 推薦方向
     - 精確比分機率
     - 雙方各自至少贏一局機率
     - 信心度
     - 投注建議
     - 主要風險

10. **賽後檢討與模型校準**
   - 如果使用者提到先前預測失準，必須改用賽後檢討模式，不要直接重新預測或替舊結論辯護。
   - 先核對實際結果、版本、先發、每局勝方、選邊、BP 與可取得的 VOD/數據來源，再比較原預測。
   - 必須拆解「方向錯誤」與「比分錯誤」：例如原預測 T1 3-2、實際 HLE 3-1，屬於勝方錯誤且至少 2 局差距的大偏差。
   - 即使勝方方向正確，只要原推薦重心是打滿、地圖大分、敗方 +1.5 或敗方至少一局，而實際為橫掃，也必須標記為「地圖分布大偏差」，不可用勝方看對淡化錯誤。
   - 將錯誤歸因到可行動類別：資料錯漏、版本/BP 判讀錯誤、近期狀態權重錯配、H2H/角色對位低估、BO5 局間調整低估、選邊價值誤判、隊伍品牌/人氣偏誤、臨場失誤與合理變異。
   - 檢討時必須指出哪些賽前假設被實際 BP 或比賽內容推翻，並提出下次預測要調整的權重或信心上限。
   - 如果出現「勝方錯 + 精確比分相差至少 2 局」的大偏差，未來同類 matchup 的預測信心度預設上限降低 10–15 個百分點，直到新的 VOD/數據證據足以解除。
   - 大偏差後，精確比分不得只押單一劇本；必須拉高對手 3-1/3-2 或 2-1 的尾端機率，並在輸出中明確說明 upset path。
   - 參考 `references/postmortem-calibration.md` 執行完整檢討。

## 必要輸出結構

每一場比賽使用以下結構：

1. **賽事資訊**
   - 賽區 / 階段 / 台灣時間日期 / 賽制 / 版本 / 資料狀態。

2. **雙方先發名單（請使用者檢查）**
   - 隊伍 A：Top, Jungle, Mid, ADC, Support。
   - 隊伍 B：Top, Jungle, Mid, ADC, Support。
   - 標記名單狀態為已確認 / 可能 / 未驗證。

3. **近期狀態與內容分析**
   - 過去一個月概況。
   - 最近 1–2 週優先觀察。
   - 對戰紀錄。
   - 主要強弱點。

4. **版本與 BP 影響**
   - 版本影響。
   - 預期 BP 優先級與 ban 壓力。
   - 選邊影響。

5. **對位重點**
   - Top / Jungle / Mid / Bottom / Support 或團隊層面的對位。
   - 說明可能左右比賽的線路與節奏槓桿。

6. **比分機率與信心度**
   - 比賽勝率。
   - 精確比分機率分布。
   - 雙方各自至少贏一局機率。
   - 信心度百分比。

7. **Stake / 盤口建議**
   - 市場觀點。
   - 公允賠率、市場賠率（十進位）或 EV 註記。
   - 推薦閘門通過/未通過理由，尤其是打滿、地圖讓分與至少贏一局。
   - 建議玩法與注碼。
   - 建議避開的市場。

8. **潛在風險**
   - 以條列列出風險。

9. **簡表總結**
   - 一張精簡表格，必須包含 `時間 (UTC+8)` 欄位，填入分析的比賽時間。

## 機率一致性檢查

- 每個系列賽的精確比分機率總和必須等於 100%。
- 比賽勝率必須等於該隊所有獲勝精確比分機率的總和。
- 某隊至少贏一局的機率，必須等於 100% 減去該隊被橫掃的機率。
- 信心度不是勝率。信心度反映證據品質、名單確定性、版本清晰度與市場一致性。
- 預測強隊或人氣隊時，必須檢查「品牌/名氣是否取代近期內容」：若推薦方向主要來自歷史聲望、明星選手或過往大賽印象，信心度需下修，並補足最近 1–2 週的 VOD/數據證據。
- BO5 精確比分若給出 3-2，必須同時說明雙方各自 3-1/1-3 的條件；若無法清楚描述對手 3-1 路徑，代表模型可能低估系列賽滾雪球與局間調整風險。
- 如果一隊近兩個系列賽出現被橫掃、前 15 分鐘持續落後、核心位置英雄池被鎖死、或對手剛打出強勢季後賽連勝，該隊「至少贏一局」不得預設高於 80%；除非能提供明確 BP/選邊/對位證據。
- BO5 橫掃機率不得機械性壓低。若有兩項以上橫掃風險因子，勝方 3-0/敗方 0-3 機率至少應進入可見區間，且輸出中需解釋為何不是打滿劇本。
- 若市場熱門 ML 約 1.45 以下或 implied probability 約 68% 以上，而模型想推薦敗方 +1.5，必須先完成市場分歧檢查；沒有兩條可驗證取圖路徑時，熱門方 3-0 機率不得壓到 10% 以下，並應避免推薦敗方地圖盤。
- BO5 剛打滿五局不得一律視為疲勞扣分；必須拆解系列賽形狀。若勝方在 1-2 落後後靠不同核心/BP 解法連拿 G4/G5，且決勝局乾淨收束，隔日分析要同時給予「手感、資訊量、局間解題」加分。
- 若 BO5 弱方 +1.5 機率高於 55% 或 Over 3.5 高於 70%，且弱方取圖路徑不是純對手失誤，精確比分分布必須把弱方 3-2/3-1 放進可見區間；熱門方勝率與信心度需同步下修，除非能提出明確的弱方收尾缺陷。

## 引用與透明度要求

- 使用網路來源時，重要事實主張需要附引用。
- 清楚區分已確認事實與推論。
- 如果資料來源衝突，賽程、版本與名單優先採用 LoL Fandom，並說明衝突。
- 如果使用者要求賽後檢討，需比較賽前預測假設與實際 BP/比賽執行，指出差異。
- 賽後檢討必須列出「下次模型修正」，且至少包含一項權重調整、一項資料檢查補強與一項信心度/注碼控管修正。

## 語氣

- 直接、分析型、實用。
- 使用台灣常見的繁體中文電競術語。
- 使用者重視深度，不要只列比分。
- 當名單、版本或 VOD 證據不完整時，避免過度自信。

## Notion 匯出

當使用者要求寫入 Notion，或環境變數 `NOTION_AUTO_PUBLISH=1` 時，完整分析或賽後檢討完成後必須依 `../shared/notion/skill-instructions.md` 執行匯出。

本模組 summary JSON 固定帶入：

```json
{
  "module": "lol-analysis",
  "sport": "LoL"
}
```

單場深度分析以每場一筆 Notion page 為預設；今日多場決策總結則可用一筆 `analysisType: "daily-summary"` 保存整份總結。賽後檢討請使用 `analysisType: "postmortem"`。
